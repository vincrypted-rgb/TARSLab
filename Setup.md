# Lab: Using the JPCERT Tool Analysis Result Sheet

> [!TIP]
> ### What is the JPCERT Tool Analysis Result Sheet?
>
> The JPCERT Tool Analysis Result Sheet (often shortened to "Tool Analysis") is a reference created by **JPCERT/CC** (Japan Computer Emergency Response Team Coordination Center). It catalogs the log artifacts generated by 49 common Windows tools.
>
> Blue teamers, incident responders, and threat hunters use it to identify lateral movement and other malicious activity within a system or network.

> The [GitHub page for the Tool Analysis Result Sheet](https://jpcertcc.github.io/ToolAnalysisResultSheet/) contains additional reports and background information that this lab may not cover.

---

## Lab Goals

The goal of this lab is to introduce the JPCERT Tool Analysis Result Sheet and its uses, while building critical blue team thinking that you can apply to investigate unknown processes.

### In this lab, you will:

- Review detailed event logs and compare them with the Result Sheet
- Determine what process was run and what the implications might be
- Learn why Tool Analysis is used in incident response and threat hunting
- Understand the Tool Analysis methodology and how to apply it elsewhere
- Use the Tool Analysis concept to identify a process not listed on the sheet

### You will need:

- A Windows VM
- The [JPCERT Tool Analysis Result Sheet](https://jpcertcc.github.io/ToolAnalysisResultSheet/)
- A little imagination (mentioned in note below)

---

> [!NOTE]
> ### Attacker → Defender Mindset
>
> In this lab, you'll briefly act as an **attacker** to generate the logs that you'll be reviewing. All attacker steps will be clearly marked.
>
> Later, you'll switch to a **defender** role. In a real incident, responders don't know what commands were run — but you will. To get the most out of this exercise, try to approach the defender phase with fresh eyes. Consider clearing your terminal or avoiding the attack section when you review logs.

## Setup

The official Tool Analysis documentation recommends having **Audit Policy enabled** and **Sysmon installed**. Here's how to set both up.

### Part 1: Installing Sysmon

#### Download Sysmon

1. Open a browser in your VM and search for **Sysmon**.
2. Click the official Microsoft Sysmon download page (it should be the top result).
3. Download the `.zip` file.

#### Extract the Files

Unzip the folder using either:
- **Extract All** in the navigation bar, or
- Right-click → **Extract All**

#### Download the Configuration File

Now we need a configuration file to make Sysmon more useful. This file filters events so we only see what matters.

1. Search for **"SwiftOnSecurity sysmon"** (or just **"Sysmon Configuration File"**) and navigate to the GitHub page.
2. Click on **sysmonconfig-export.xml**.
3. Click the **Download raw file** button on the right side of the screen.

> [!CAUTION]
> Save the configuration file **in the same folder** where you extracted the Sysmon files — not inside the `.zip`, and not in your Downloads folder. If you forget, you'll just need to move it later, but saving it correctly now saves a step.

#### Install Sysmon

1. Close your browser.
2. Open **Command Prompt as an administrator**.
3. Navigate to the folder where you extracted Sysmon.

> [!TIP]
> **Command Prompt Quick Reference:**
> - `dir` — Lists files and folders in your current directory
> - `cd` — Shows your current directory path
> - `cd [folder name]` — Moves into a folder (case-sensitive)
> - `cd ..` — Moves up one directory level
> - `whoami` — Prints the current username

4. Run the installation command:

```cmd
.\sysmon64.exe -accepteula -i sysmonconfig-export.xml
```

*(If you're on a 32-bit system, use `sysmon.exe` instead of `sysmon64.exe`.)*

> [!NOTE]
> If you renamed your configuration file, replace `sysmonconfig-export.xml` with your filename.

#### Verify Sysmon is Running

1. Open **Event Viewer** (search for it, or press `Windows+R` and type `eventvwr.msc`).
2. Navigate to **Applications and Services Logs > Microsoft > Windows > Sysmon > Operational**.
3. If you see events, it's working. If the Sysmon folder isn't there, try refreshing or restarting Event Viewer. If it still doesn't appear, the installation failed — review the steps above.

### Part 2: Enabling Audit Policy

There are two ways to enable Audit Policy — through **Group Policy** (GUI) or **PowerShell** (command line). We'll use Group Policy here since it's more beginner-friendly.

> [!NOTE]
> The JPCERT report wasn't specific about which Audit Policy settings they used. We'll cover the bare minimum here, but feel free to enable more.

#### Group Policy Method

1. Press `Windows+R`, type `gpedit.msc`, and press Enter.

*(Screenshot: Group Policy window)*

2. Navigate to **Local Computer Policy > Windows Settings > Security Settings**.

> [!TIP]
> Alternatively, you can type `secpol.msc` into Run to go directly to Security Settings. We're using `gpedit.msc` because we'll need Administrative Templates later.

3. Click **Advanced Audit Policy Configuration**, then expand **System Audit Policies - Local Group**.

4. Browse each subsection and enable the policies you want. Use the **Explain** tab in each policy to understand what it logs and how verbose it is.

**At minimum, enable these under Detailed Tracking:**
- **Audit Process Creation** — Success and Failure
- **Audit Process Termination** — Success and Failure

**Important:** Click **Apply** before leaving each policy window — changes won't save automatically.

> [!TIP]
> ### PowerShell Shortcut (For the Impatient)
>
> If you'd rather skip all the clicking, open PowerShell **as an administrator** and run each of these on separate lines:
>
> ```powershell
> auditpol /set /subcategory:"Process Creation" /success:enable /failure:enable
> auditpol /set /subcategory:"Process Termination" /success:enable /failure:enable
> ```
>
> That's it. Done.
>
> *(But if you're new to Windows security, the Group Policy method above is worth doing at least once so you know where everything lives.)*

> [!TIP]
> ### Enable Command Line Logging
> For more detailed logs — especially if an attacker is connecting remotely via WinRM or RDP — enable the **Include Command Line** policy.
>
> Navigate to **Administrative Templates > System > Audit Process Creation** (if you don't see Administrative Templates, you opened `secpol.msc` instead of `gpedit.msc`). Set it to **Enabled**, then click **Apply**.

---

### Part 3: Increasing Log Sizes

Before we start generating logs, increase the log sizes so we don't lose data.

#### For the Security Log:

1. Open **Event Viewer**.
2. Expand **Windows Logs**.
3. Right-click **Security** → **Properties**.
4. Under **Maximum Log Size (KB)**, enter a new size. (The max is 4,194,240 KB / 4GB.)
5. Select **Overwrite as needed (oldest events first)**, then click **Apply**.

#### For the Sysmon Log:

1. Navigate to **Applications and Services Logs > Microsoft > Windows > Sysmon > Operational**.
2. Right-click **Operational** → **Properties**.
3. Increase the size and set it to **Overwrite as needed**.
4. Click **Apply**.

## Troubleshooting

| Issue | Solution |
| :--- | :--- |
| `gpedit.msc` not found | You're on Windows Home edition. Upgrade to Pro or use the PowerShell method instead. |
| Sysmon folder missing in Event Viewer | Refresh Event Viewer or restart the Sysmon service. If that doesn't work, reinstall Sysmon. |
| Config file not found during install | Make sure the `.xml` file is in the same folder as `sysmon64.exe`. |
| Command prompt says access denied | Make sure you're running Command Prompt **as an administrator**. |
